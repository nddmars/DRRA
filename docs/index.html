Resilience Forge (DRRA)

Distributed Resilience & Recovery Architecture — The industry standard for architecting, testing, and validating ransomware-proof ecosystems through automation and AI.

🔵 WALL — PREVENT 🟡 SQUAT — SURVIVE 🔴 GRAB — CONTROL 🔥 FORGE — SIMULATE

What is DRRA?

Resilience Forge is an open-source "Assume Breach" laboratory that provides a controlled playground where security engineers can build, test, and validate ransomware defenses. Unlike traditional security tools that focus on prevention alone, DRRA implements the WALL-SQUAT-GRAB framework — a three-pillar approach designed to detect threats, contain their blast radius, and recover with zero data loss.

The system is built around the concept of the Defensibility Index (DI) — a 0–100 score measuring how well your infrastructure would survive a real ransomware attack. Higher is better.

The WALL-SQUAT-GRAB Framework

🔵

WALL — PREVENT

Component: Vigil

AI-driven behavioral detection identifies threats before encryption spreads. Entropy analysis, mass modification detection, and lateral movement tracking using ML-based pattern recognition.

🟡

SQUAT — SURVIVE

Component: Shield (Contain)

SOAR automation and rapid response limits blast radius. Dynamic micro-segmentation isolates compromised systems in <30 seconds, preventing lateral spread.

🟢

GRAB — CONTROL

Component: Shield (Recover)

Immutable backups ensure recovery capability. Verified backup restoration validates data integrity with automated failover to known-good state.

Components

Four core components implement the WALL-SQUAT-GRAB framework. Each maps to specific defense pillars and exposes a REST API consumed by the Dashboard and automation workflows.

🔥 The Forge — Testing & Simulation

Directory: forge/  |  Service: backend/services/forge_service.py

Safe, controlled simulation of ransomware behaviors for testing all three pillars (WALL, SQUAT, GRAB) without deploying real threats. The Forge is the "Red Team in a box."

Sub-components

Honeypot Architect

  • Generates realistic file structures (PDF, Excel, SQL dumps)
  • Configurable file size: 1–100 MB per file
  • Modified timestamps blend with real data
  • Acts as detection tripwires

Resilience Payloads

  • Entropy spike mimicking real encryption
  • File modification patterns matching real threats
  • Non-destructive (reads files, no actual encryption)
  • Configurable intensity (0.0–1.0)

Identity Squatting Tests

  • Kerberos lateral movement simulation
  • Unauthorized ticket request patterns
  • Credential reuse detection
  • Domain enumeration attempts

Payload Types

  • honeypot — Trap file generation
  • mass_modification — Bulk file changes
  • encryption_sim — Entropy injection
  • lateral_movement — Network scan simulation

👁 Vigil — Detection Engine (WALL/PREVENT)

Directory: vigil/  |  Key file: vigil/detector.py

Detect threats before they cause damage through AI-driven behavioral analysis. Vigil implements the WALL pillar — the first line of defense that identifies ransomware indicators and feeds the Shield for automated response.

Sub-components

Behavioral ML Engine

  • Mass modification detection (>15% files in 60s)
  • Entropy analysis (>0.85 indicates encryption)
  • Lateral movement tracking via process chains
  • VSS (Volume Shadow Copy) abuse detection

Immutable Telemetry Pipeline

  • Write-once logging: Vector/Fluentd → Kafka → MinIO
  • Object Lock prevents deletion (365-day retention)
  • Tamper-proof: HIPAA, PCI-DSS, SOC2 compliant
  • All events timestamped and hash-verified

LLM Integration (Gemini 2.5 Flash)

  • Plain-English attack summarization
  • Recommended hardening actions
  • Defensibility gap identification
  • Contextual threat intelligence

🛡 The Shield — Response & Recovery (SQUAT/GRAB)

Directory: shield/  |  Key file: backend/services/shield_service.py

Rapidly contain threats (SQUAT) AND restore operations from verified backups (GRAB). The Shield implements both survival and control pillars.

SQUAT — Dynamic Micro-Segmentation

  • Automated VLAN isolation via API
  • Network quarantine in <30 seconds
  • Blocks lateral movement immediately
  • Rollback capability for false positives (<5s)

MTTC target: <60 seconds  |  Isolation: 100%

GRAB — Immutable Object Locking

  • MinIO Compliance Mode activation
  • Legal hold on forensic buckets
  • Prevents even admin deletion
  • Tamper-proof backup verification

GRAB — Recovery Orchestration

  • Snapshot restoration from verified backups
  • Credential revocation automation
  • Block-level incremental restores
  • Parallel recovery threads (default: 8)

MTTR target: <15 minutes  |  Data loss: <0.1%

GRAB — Forensic Curation

  • Automatic evidence preservation
  • Chain of custody maintained
  • Attack artifacts captured
  • 90-day default retention

Detection Engine

Vigil's detection engine combines ML-based behavioral analysis with rule-based pattern matching and LLM-powered summarization to identify ransomware with high confidence and low false positive rates.

Detection Patterns

📈

Mass File Modification

Threshold: >15% files modified in 60s
CRITICAL

Monitors file modification rates across watched directories. Ransomware typically encrypts large batches of files in rapid succession, producing a distinctive modification spike.

💡

Entropy Analysis

Threshold: Shannon entropy >0.85
CRITICAL

Calculates Shannon entropy (-∑ p·log₂(p)) on file content. Encrypted files approach maximum entropy (1.0). Benign files typically score 0.3–0.7.

🗂

VSS Shadow Copy Deletion

Trigger: vssadmin / wmic / PowerShell VSS delete
CRITICAL

Detects Volume Shadow Copy deletion attempts via vssadmin, wmic shadowcopy delete, or PowerShell Win32_ShadowCopy calls — prevents attacker recovery sabotage.

🔗

Lateral Movement

Trigger: NTLM reuse + auth spike / PsExec / WMI spawn
HIGH

Detects credential reuse over SMB (NTLM logon type 3), PsExec-style remote execution, WMI process spawning, and authentication failure spikes (>10 in 60s).

🔑

Privilege Escalation

Trigger: SeDebugPrivilege / UAC bypass / kernel driver
CRITICAL

Detects token impersonation (SeDebugPrivilege, SeImpersonatePrivilege), UAC bypass via fodhelper/eventvwr, suspicious kernel driver installation, and Office app spawning shells.

DevSecOps Pipeline — Detection as Code

DRRA treats detection rules as first-class code — versioned, reviewed, tested, and deployed through a CI/CD pipeline. This section explains how Sigma rules power the detection engine and how the full policy-as-code ecosystem works.

What is Sigma?

  • Industry-standard detection rule format — like Snort rules for SIEMs. Write once, deploy everywhere.
  • Plain YAML — readable and reviewable by security engineers AND developers, not just SIEM specialists.
  • Platform-agnostic — one Sigma rule compiles to Splunk SPL, Elasticsearch DSL, Azure Sentinel KQL, IBM QRadar, and Google Chronicle YARA-L.
  • Part of the DevSecOps pipeline — rules are treated like code: version-controlled, pull-request reviewed, and CI-validated before merge.

Sigma in DRRA

Rules Library

  • DRRA ships with 4 Sigma rules covering all major ransomware phases
  • Rules live in drra-policies/sigma/ — versioned with the codebase
  • Each rule maps to a MITRE ATT&CK technique
  • VIGIL consumes compiled rules at runtime for real-time detection

CI/CD Integration

  • sigma-cli validates rule syntax on every commit
  • pytest runs behavioral tests against known-good and known-bad samples
  • Failed validation blocks merge — no broken rules reach production
  • Auto-deploy to VIGIL on merge to main

Step-by-Step Setup

Clone the Repository

git clone https://github.com/resilience-forge/drra.git
cd resilience-forge
cp .env.example .env

Start All Services

docker-compose up -d

Wait 30–60 seconds for all containers to become healthy. Verify with: docker-compose ps

Install Python Dependencies

pip install -r requirements.txt
pip install pytest pytest-asyncio pytest-cov flake8

Start the Backend API

cd backend
uvicorn main:app --reload --host 0.0.0.0 --port 8000

API available at http://localhost:8000 — Swagger UI at http://localhost:8000/docs

Verify System Health

curl http://localhost:8000/health | python -m json.tool

Run Your First Simulation

# 1. Deploy a honeypot
curl -X POST http://localhost:8000/api/v1/forge/deploy \
  -H "Content-Type: application/json" \
  -d '{"name":"demo_attack","payload_type":"mass_modification","target_path":"/tmp/honeypot","duration_seconds":60,"intensity":1.0}'

# 2. Watch Vigil detect it
curl "http://localhost:8000/api/v1/vigil/events?threat_level=critical" | python -m json.tool

# 3. Check the Dashboard
open http://localhost:7700

Run the Test Suite (The Gauntlet)

# Run all tests
pytest tests/ -v

# Run full end-to-end scenario
pytest tests/test_gauntlet.py::TestGauntletScenarios::test_complete_ransomware_scenario -v

# Run with coverage
pytest tests/ --cov=backend --cov-report=html

Build pass requirements: data loss <0.1%, MTTC <60s, all logs immutable, detection confidence >90%
